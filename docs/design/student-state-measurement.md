## §1.1 The event log: two classes of event (actions and signals)

Everything the system measures is a fold over one append-only event log. The log
holds two kinds of event, and the difference is load-bearing:

- **Actions** — semantic moves the child makes (place block, submit answer, write
  a fraction). The **reducer folds actions into game state.** Every action input
  modality (drag, tap, type, voice-answer, handwriting) produces these.
- **Signals** — passive observations about the child, not moves (affect/attention
  from the camera; idle; place-remove oscillation; latency). Signals are appended
  to the log but the **reducer treats them as no-ops on game state.** They are
  read by the student model and the Tier-2/Tier-3 brain only.

This is the rule that keeps a noisy sensor from corrupting the game: **a signal
can change pacing and support; it can never change game state, what is true, or
mastery.** The affect camera is a signal, full stop. So is "the child looks
frustrated." Neither can place a block, answer a question, or certify a skill.

```
Event = Action{ type, payload, modality, t, actor }     // → reducer → game state
      | Signal{ type, payload, confidence, t, actor }    // → model / brain only
```

`actor ∈ {human, synthetic:persona}` so a synthetic learner (companion §8) can emit
both classes, including synthetic affect — which is why the same measurement code
path serves real and synthetic students, and why synthetic evidence is credible.

### What gets recorded per attempt

One problem produces a **burst** of events, e.g.:

```
problem_present, block_place×N, idle, hint_shown{rung}, self_correct,
answer_submit{value}, judged{correct, latency}
```

Plus asynchronous signal events (affect at ~1–2 Hz, idle timers). The measurement
layer (§4.7.4) collapses each burst into one **Observation** at the problem
boundary. Correctness, latency, scaffold level, hint rung, the submitted answer
value, and self-corrections are all recorded — not just right/wrong.

## §4 The Student-State / Mastery Model (how we measure performance)

This is the heart of the `hyper_responsive_ui` brief: "explain what the learner
can now do that they could not do before," and guard against false-positive
mastery. Per skill node we maintain four dimensions. The certifying gate (§4.5) is
deterministic; the LLM brain (companion §5) reads these numbers but cannot overrule
them.

> These four dimensions are the *shipping floor*. §4.7 reframes them as the
> 2-state special case of a per-node Hidden Markov Model and reserves a staged
> upgrade to a richer **factorial HMM** (adding a `misconception` state, a vector
> emission, and strategy/engagement chains) behind the same `MasteryEstimate`
> interface. Read §4.1–4.6 as what we build first; §4.7 as where it goes.

### §4.1 Accuracy — Bayesian Knowledge Tracing (BKT)

Standard, inspectable, defensible. Per node, four parameters
`{P_L0, P_T, P_S, P_G}`. Placeholder defaults, to be tuned by the synthetic
harness (companion §8): `P_T = 0.20, P_S = 0.10, P_G = 0.20`.

**Initialization (cold start).** Before any observation:

```
P_known(node) = clamp( base_prior(node) + Σ_{p ∈ prereqs} w · (P_known(p) − 0.5),
                       [0.05, 0.85] )
```

(BKT is the 2-state Hidden Markov Model; §4.7 makes that framing explicit and
reserves the richer-emission / more-states upgrade.)

`base_prior` defaults to `P_L0`; `w` (default 0.3) lets a strong prereq raise the
prior, which is what produces **skip-ahead**. Clamp to `[0.05, 0.85]` so no node
starts in an absorbing 0/1 state. The very first kitchen recipe is match-a-height
(no arithmetic), so the first real arithmetic observation always lands after some
baseline signal exists.

**Update after each observation** (correct BKT; the posterior, then the learn step):

```
correct:    P(known|obs) = P_known·(1−S) / [ P_known·(1−S) + (1−P_known)·G ]
incorrect:  P(known|obs) = P_known·S     / [ P_known·S     + (1−P_known)·(1−G) ]
then learn: P_known'      = P(known|obs) + (1 − P(known|obs))·T
```

(The `incorrect` denominator uses `(1−G)` — the chance of answering wrong while
*not* knowing the skill. This is the corrected equation.) Clamp `P_known'` to
`[0.01, 0.99]`.

### §4.2 Fluency — the philosophy doc's "speed and directness"

Per node, computed over the last `N` correct answers (require `N ≥ 5` before the
fluency gate is allowed to evaluate at all):

```
Fluency OK  ⟺  median_latency(last N correct) ≤ age_band(skill)
            AND latency slope over the window ≤ ε   (ε small positive; not strictly 0,
                                                     to tolerate noise)
```

`age_band(skill)` is a per-skill latency target. It is uncalibrated until a pilot
(Open Q1), so in early milestones **fluency is a soft/advisory gate**: it informs
the brain but does not block the mastery gate. It becomes a hard gate once
`age_band` is calibrated. (This resolves the soft-vs-hard inconsistency: §4.5
treats fluency as soft until calibration, hard after.)

### §4.3 Scaffold-independence

```
Independent ⟺ ≥ 2 correct at scaffold level ≥ L3 (numbers only),
              on ≥ 2 distinct generated problems, all with hint_rung == 0
```

Two observations, not one, so a single lucky correct cannot pass it. Guards
"succeed only while the UI does the reasoning for them." (Scaffold ladder L0–L4 and
the hint ladder H0–H4 are defined in the companion doc §3.)

### §4.4 Transfer

```
Transfer    ⟺ ≥ 2 correct on ≥ 2 structurally distinct surface_forms,
              at low scaffold, hint_rung == 0, latency in band
```

Structurally distinct = not just different numbers; a different shape of problem
(reversed, embedded in a new recipe, different denominator relationship). This is
the brief's required transfer moment, built into every node's exit, and the thing
the synthetic "memorizer" persona is designed to fail (companion §8).

### §4.5 The Mastery Gate (deterministic; the LLM cannot set this)

```
MASTERED(node) ⟺ P_known ≥ 0.95
              AND Scaffold-independent (§4.3)
              AND Transfer passed (§4.4)
              AND Fluency OK            // soft (advisory) pre-calibration; hard after
```

`MasteryEstimate = { P_known, fluency_stats, max_scaffold_passed,
transfer_passed, hint_dependence, last_retention_probe }`. The whole student
state is `{ node_id → MasteryEstimate }` plus a short-term behavior buffer
(recent signals, including the smoothed affect/attention state). **`MasteryEstimate`
is the stable interface to the rest of the system** — the brain, gate consumers, UI,
and harness all read it; every model upgrade (§4.7) swaps the inference behind it.

**Affect is not in the gate, by construction.** Notice no affect term appears in
`MASTERED`. A child who looks confident has not mastered anything; a child who
looks frustrated may have. Affect changes how hard we push and how much we support
(companion §3.1, §5; this doc §6.1); it is never evidence of knowing. This is the
line that keeps the camera honest against the brief's "don't hide uncertainty
behind confident UI."

### §4.6 Decay / retention

A `MASTERED` node is scheduled a **spaced retention probe** (a low-scaffold,
hint-free item after a delay). Failing it demotes the node: clear
`transfer_passed`, drop `P_known` below threshold. A demoted node can be re-opened
by the next wall. This is what makes the counter-metric "false-positive mastery
rate" (% of mastered nodes that later fail a delayed probe) measurable instead of
hypothetical.

### §4.7 The factorial-HMM framing (reserved design, staged upgrade)

The four dimensions in §4.1–4.4 are, today, computed by four separate pieces of
machinery (a BKT update, a fluency EWMA, a scaffold counter, a transfer counter)
and then AND-ed at the gate (§4.5). That works, ships, and is defensible. But it
leaves a richer model on the table, and this section reserves the seam for it —
**no code yet**, same staging discipline as the deterministic→LLM brain swap
(companion §9).

#### §4.7.1 Why an HMM: BKT already is one

**Bayesian Knowledge Tracing is a Hidden Markov Model** — the 2-state special case.
Mapping it explicitly:

- **Hidden state** = `{ known, not-known }` — the unobservable knowledge state.
- **Transition** = `P_T` (`not-known → known`); within-session forgetting is the
  transition BKT zeroes out, and §4.6 decay is exactly that transition re-introduced
  across sessions.
- **Emission** = correct / incorrect, related to the hidden state by `P_S` (slip:
  known but wrong) and `P_G` (guess: not-known but right).
- **Inference** = the §4.1 update is the HMM forward algorithm on two states.

So "model the evolving probability of knowing over time" is already what we do. The
upgrade is not a new paradigm; it is **a richer emission and more hidden states**,
which is precisely what lets the model use the multimodal signal stream that plain
BKT throws away (it only consumes one bit, correct/incorrect).

#### §4.7.2 The state set is three independent chains, not one list

The interesting hidden states are not mutually exclusive — a child can be *fluent
and bored*, or *guessing and anxious*. A single flat categorical state cannot
represent that, so the target is a **factorial HMM**: parallel hidden chains that
jointly produce each observation.

**Chain A — Knowledge (the spine; the mastery gate reads ONLY this chain):**

```
unknown          absent knowledge → diffuse, non-signature errors
misconception    an actively wrong rule → a SPECIFIC, predictable wrong answer
acquiring        fragile; sometimes right, effortful
known-effortful  reliably right but slow ("knows-but-slow")
fluent           right and fast/automatic
```

**Chain B — Response strategy (how *this* correct answer was produced):**

```
genuine          worked it
guessing         high-variance correctness, fast, no working
pattern-matching right on a seen format, fails on a changed surface form
hint/scaffold-carried   correct only with support present
```

**Chain C — Engagement (disposition; fed by affect + behavior):**

```
engaged · frustrated · bored · avoiding
```

Corrections this framing makes to the informal list we brainstormed:

- **"hint-dependent" is not a hidden state — it is observed.** The hint rung is in
  the event log (companion §3.2). It is an *input* that helps infer Chain B
  (`hint-carried`), not something to predict. Don't spend a hidden state on a
  measured quantity.
- **"knows-but-slow" is not its own state — it is `known-effortful`**, and *latency*
  is the observation that separates it from `fluent`. The hidden thing is "do they
  know it"; speed is how we read effortful-known vs. automatic-known.
- **`misconception` is the highest-value addition** and was missing. `unknown`
  predicts scattered errors; `misconception` predicts a *clustered, signature* wrong
  answer (the room docs name these: 2/7 + 3/7 → 5/14 "added the bottoms"; 1/2 + 1/3
  → 2/5). It earns a hidden state because it emits differently from `unknown`, and
  the brief explicitly cares about "which misconceptions survive instruction."

Chains B and C **are the synthetic personas** (companion §8): the guesser *is*
Chain-B `guessing`; the memorizer *is* `pattern-matching`; the anxious learner *is*
Chain-C `frustrated`. That correspondence is what lets the harness both test and
calibrate this model (§4.7.5).

#### §4.7.3 The affect firewall (consistency with §4.5)

The factorial structure keeps a promise the rest of the design makes: **affect
never gates mastery.** Affect feeds Chain C and may help *disambiguate* the others
(frustrated + wrong = real struggle, not carelessness; bored + fast-correct = ready
to skip). But the **mastery gate (§4.5) reads only the Chain-A posterior.** Affect
informs inference; it never certifies knowledge. This is the §4.5 "affect is not in
the gate" rule, restated at the model level.

#### §4.7.4 The event → inference pipeline (the wiring)

§1.1 defined the event *log* and "state = fold over events," but not the fold
itself. The fold is five named components, currently undefined and required before
any HMM (or even the richer BKT) can run:

```
raw events ─[1]─► attempt ─[2]─► Observation vector ─[3]─► routed to skill(s)
                                                              │
                                                            [4] HMM step
                                                              ▼
                                                       MasteryEstimate → gate/brain
```

1. **Segment — the inference tick is the *attempt*, not the raw event.** One problem
   emits a burst (`problem_present, block_place×N, idle, hint_shown, self_correct,
   answer_submit, judged`). The model steps **once per completed attempt**, at the
   `JUDGED → policy call` boundary (companion §2) — which already aligns with "brain
   consulted only at boundaries." This aggregation layer must exist; it does not today.

2. **Featurize — the emission is a *vector*, and must include the actual answer.** To
   distinguish `misconception` from `unknown`, the wrong answer is fingerprinted
   against known error signatures, so:

   ```
   Observation = { correct, answer_value, error_signature,
                   latency, hint_max_rung, self_corrections,
                   scaffold_level, affect_window }
   ```

   The "events → this vector" function is the heart of the wiring. Plain BKT dodges
   it (emission = one bit); the moment we go multi-signal it is the real work.

3. **Route — credit assignment across the DAG is unsolved.** BKT assumes one skill
   per item; the prerequisite DAG (companion §1) breaks that. A wrong unlike-
   denominator answer is evidence about `ADD_UNLIKE_DEN` *or* the underlying
   `ADD_SAME_DEN` step *or* counting beneath it. Which chain(s) an observation
   updates, and how blame is split, must be decided. (First-pass option: update the
   binding node, and propagate a discounted update to a prereq only when the
   `error_signature` implicates it.)

4. **Update — advance the posteriors.** The HMM step over the (factored) hidden state
   given the Observation, producing the `MasteryEstimate`. The gate reads Chain A;
   the policy/brain (companion §5) may read all three.

5. **Window — continuous signals vs. discrete attempts.** Affect/attention arrives
   ~1–2 Hz, asynchronously; attempts are discrete. A windowing rule must bucket the
   affect stream and attach the right slice (`affect_window`) to each attempt's
   Observation. Undefined today.

#### §4.7.5 The harness as calibrator (not just red-team)

The cost of the richer model is **parameters we cannot calibrate without data** — a
5-state chain with a vector emission has many more than BKT's four-per-skill, and
inventing emission distributions we cannot validate is exactly the brief's "hide
uncertainty behind confident UI" failure. The unlock: the **synthetic personas
(companion §8) are generative models of Chains B/C**, so running them produces
labeled synthetic data to *estimate the HMM's emission/transition parameters before
any real child*. The harness stops being only a critic and becomes the
parameter-estimation ground truth. This is a concrete reason the two challenger
projects belong together.

#### §4.7.6 Staged upgrade path (what actually gets built, when)

- **Floor (ships now):** 2-state per-skill BKT (§4.1) + the three separate dimension
  trackers (§4.2–4.4), AND-ed at the gate. Defensible, inspectable, brief-friendly,
  and the harness's comparison baseline. **`MasteryEstimate` is the stable interface.**
- **Upgrade 1 (highest value, lowest cost): add `misconception` to Chain A** and the
  `error_signature` featurizer (pipeline step 2). Catches the exact false-positive
  the room docs target; small state increase.
- **Upgrade 2: enrich the emission** from one bit to the full Observation vector
  (latency, hint rung, scaffold) — folds §4.2–4.4 into one inference instead of four
  hand-built gates.
- **Upgrade 3: add Chains B and C** (strategy, engagement) as parallel factors,
  calibrated by the harness (§4.7.5).

Each upgrade swaps the inference behind the same `MasteryEstimate` interface — the
gate (§4.5), the policy (companion §5), the UI, and the harness (companion §8) are
untouched. This is the companion §9 "swap one implementation of one interface"
discipline applied to the model.

## §6.1 The affect/attention signal (low-trust, corroborating)

The camera emits a **smoothed** affect/attention state, not per-frame labels
(per-frame is noise). Sampled ~1–2 Hz and rolled into:

```
AffectState { engagement: high|med|low,
              valence:    frustrated|neutral|delighted,
              attention:  on_task|drifting|away,
              confidence: 0..1 }     // emitted as Signal events (§1.1)
```

Rules that keep it from being a gimmick (the brief lists "camera without a clear
learning benefit" and "false confidence from sensors" as failure modes):

- **Corroboration before action.** Affect alone never triggers anything
  consequential. Frustration acts only if a behavioral signal agrees (errors,
  long idle, oscillation). Boredom-driven faster-fade acts only if mastery
  evidence already supports the fade. Affect *raises a hypothesis*; behavior
  *confirms* it.
- **It maps to your pedagogy.** The curriculum deliberately induces frustration
  at the wall. Affect is how the system distinguishes **productive struggle**
  (push on) from **shutdown** (re-scaffold, slow down) — the exact line the
  philosophy doc warns about ("expose the weakness, not wear the child down") —
  and how it confirms the **felt payoff** on the return (delight on the recipe
  that used to stump them).
- **Never gates mastery.** Affect changes pacing and support only (§4.5, §4.7.3).
- **Privacy.** Affect inference runs **on-device**; only derived labels enter the
  event log; **no raw video is stored or transmitted.** (Also shrinks data risk
  for a product aimed at children.)
- **It must earn its place.** Tracked by a counter-metric (companion §Success): does
  affect improve outcomes over behavioral signals alone, or just add noise? If it
  does not beat behavior-only, it gets cut.

## False-positive guards (woven through the measurement)

The brief makes guarding against shallow success non-negotiable. Each guard maps to
a specific recorded signal and a specific dimension above:

- **Guessing / luck:** BKT `P_G`; the ≥2-observation gates (§4.3/4.4); too-fast-
  correct (latency below a plausible-compute floor) forces a transfer probe.
- **Pattern-matching:** correct on a repeated format but failing a changed
  `surface_form` → the transfer dimension catches it at the gate. (In the §4.7
  upgrade this becomes an explicit Chain-B hidden state rather than a gate check.)
- **Misconception:** a clustered, signature wrong answer (e.g. 2/7+3/7→5/14) is
  distinguished from diffuse `unknown` errors by the `error_signature` featurizer
  (§4.7.4) — the highest-value model upgrade (§4.7.6).
- **Hint-dependence:** measured (companion §3.2); independence/transfer require
  `hint_rung == 0`.
- **Click-through / avoidance:** idle and rapid-submit flagged by Tier-2 (companion
  §6); these are exactly the behaviors the synthetic personas exercise (companion §8).
