## Problem Statement

Build the fraction-adding game from `app_philosophy.md` so that it satisfies the
`hyper_responsive_ui` brief (the interface is part of the tutoring; it guides,
assesses, remediates, and knows when the learner is genuinely ready), and is
structured so the `synthetic_challenger` red-team harness can later stress-test
it without a rewrite.

The pedagogy is fixed by the philosophy doc: a kitchen hub (the goal: cook with
mom), and a set of skill "rooms" you are exiled to only when the kitchen throws a
problem your current tools cannot handle cleanly. Mastery is **speed and
directness**, not eventually-correct.

## What Makes This Work (the core thesis)

One architectural decision carries the whole project:

> **All interaction flows through a single `action → reducer → state → event-log`
> core, fully decoupled from input device and from rendering.** The UI emits
> actions and renders state. The mastery model is a pure fold over the event log.
> A synthetic learner emits the same actions headlessly.

Consequences:

- **Multimodality is additive.** Drag, tap, typing, spoken answer, handwriting
  are all just different things that produce the same `action`. Adding one never
  touches the engine.
- **The mastery model is replayable and inspectable.** Student state is
  `reduce(events)`. The brief forbids "hiding uncertainty behind confident UI";
  an inspectable fold plus a recorded decision-rationale (§5) is the opposite.
- **The synthetic_challenger is the same API, different actor.** A persona is a
  policy that consumes `problem_present` events and emits action events. No
  special hook. This is the cheap way to satisfy "leave space for
  synthetic_challenger," and it is why the synthetic evidence is credible: a fake
  student runs the *same code path* as a real one.

Everything below hangs off this. If the seam is clean, the rest is content and
tuning. If it leaks (mastery logic reading mouse coordinates, a room hard-coding
its own progression, the LLM mutating state directly), both the responsive UI and
the harness become rebuilds.

### 1.1 Two classes of event: actions and signals (summary; full detail in measurement doc)

The single event log holds two kinds of event, and the distinction is load-bearing
for the engine thesis above:

- **Actions** — semantic moves (place block, submit answer, write a fraction). The
  **reducer folds actions into game state.**
- **Signals** — passive observations (affect/attention, idle, oscillation, latency).
  The **reducer treats them as no-ops on game state**; only the model/brain read them.

The rule: **a signal can change pacing and support; it can never change game state,
what is true, or mastery.** `actor ∈ {human, synthetic:persona}` so a synthetic
learner (§8) runs the same code path. Full event schema, the per-attempt burst, and
how events become a `MasteryEstimate` are in **`student-state-measurement.md` §1.1**.

## Constraints (decided this session)

- **D1 — Adaptive from day one.** A real per-skill mastery model drives
  sequencing, not completion bars.
- **D2 — Full curriculum.** Kitchen hub + 4 fraction rooms (R1–R4). The root
  learning room is `ADD_SAME_DEN`.
- **D3 — Full multimodal set.** *Action* inputs: drag-to-stack, tap/type,
  handwriting. *Signal* inputs: an affect/attention camera (emotion, engagement,
  on-task attention) plus the passive behavioral signals (idle, oscillation).
  Outputs: visual workspace, animated feedback, spoken tutor. All in scope.
  (See §1.1 for the action-vs-signal distinction; it governs what affect may do.)
- **D4 — Everything, fully built (kick-ass demo).** No feature is cut. "Engine
  first" is retained only as a *build-dependency order* (you cannot build a room
  before the engine that runs it), not as a scope reduction.
- **D5 — Deterministic brain (amended 2026-05-31; LLM removed for scope).** Tier-3
  pedagogical decisions are made by a **deterministic policy** (§5.4) reading the
  structured student state, bounded by legal-move enumeration and the exact verifier.
  (Originally an LLM over structured state; cut as too much scope. The guardrails remain
  as plain determinism.)

## Premises (agreed)

1. **The engine is the asset; everything else is replaceable.**
2. **Build in dependency order.** Engine → one complete loop → fan out rooms and
   modalities. Nothing dropped; just sequenced so something always runs.
3. **Mastery is multi-dimensional; completion is not mastery.** Accuracy +
   fluency + scaffold-independence + transfer, and the certifying gate is
   deterministic (the LLM never declares mastery).
4. **The "wall" is a diagnostic event, not a script trigger.** Route to the most
   upstream unmastered prerequisite of the blocked skill.
5. **The interface changes only on demonstrated state change**, never on a timer
   or to show a component. Some things refuse to auto-change.

## 1. The Skill Graph (curriculum as a DAG)

A directed acyclic graph of skill nodes rooted at the goal. This is the literal
structure the philosophy doc describes ("a mental graph rooted at the goal").

```
                         [GOAL: cook with mom]   ← kitchen hub (not a skill node;
                                   │                the place walls are thrown)
                                   │ core fraction layer
                                   ▼
              R1 ADD_SAME_DEN ──► R2 ADD_UNLIKE_DEN
                                          │
                          ┌───────────────┤
                          ▼               ▼
                    R3 SIMPLIFY    R4 IMPROPER_TO_MIXED
```

**Edge semantics.** Solid edges are *hard prerequisites* (you cannot be assessed
on a node until its prereqs are mastered enough to not be the binding constraint).
`R3 SIMPLIFY` and `R4 IMPROPER_TO_MIXED` are **both** successors of
`R2 ADD_UNLIKE_DEN` and are **not** hard prerequisites of each other: you can turn
9/7 into 1 2/7 without simplifying, and you can simplify without converting. The
philosophy doc *sequences* simplify before improper in the kitchen's recipe
progression (curriculum order), but that is a recipe-ordering choice, not a graph
edge. Keep them as parallel successors of R2.

Each **node** carries:

```
SkillNode {
  id                 // e.g. "ADD_UNLIKE_DEN"
  prereqs[]          // hard-prerequisite edges into this node
  mastery: MasteryEstimate   // §4 — the whole point
  scaffold_ladder[]  // ordered scaffolds for this skill (§3)
  problem_generator  // deterministic, parameterized, with an exact verifier
  transfer_forms[]   // ≥2 structurally distinct surface forms (§4.4)
  bkt_params         // {P_L0, P_T, P_S, P_G} (§4.1)
}
```

Sub-skills (the philosophy doc's "recognize what digits are divisible by what",
"translate block representation to numeric") start **folded into their node** and
are split into their own node only when a wall ever needs to route to them
(premise 4; the philosophy doc's "a concept gets a room only once a gap is
actually constraining progress"). Start coarse: 4 nodes. Split on demand.

Edges are prerequisites, not a fixed path. The path a child takes is chosen at
runtime by the brain (§5) from their mastery estimates. That is what makes it
adaptive rather than a content path.

## 2. Game States & Sequencing (the state machine)

Two nested levels. The **session machine** (where the child is) and, inside any
room or kitchen attempt, the **problem machine** (the lifecycle of one problem).

### Session machine

```
            ┌───────────────────────────────────────────────────────────┐
            ▼                                                             │
       [KITCHEN] ──present recipe──► attempt                              │
            ▲                          │                                  │
            │                ┌─────────┴──────────┐                       │
            │          success                 WALL_HIT                   │
            │        (gate passed)        (binding skill found, §5)       │
            │              │                     │                        │
   demonstrate skill       │                     ▼                        │
   on the exact recipe     │               [ROOM: node]                   │
   that stumped them ◄─────┘                     │                        │
            │                          drill, fade scaffold,             │
            │                          until MASTERY_GATE (§4.5)          │
            │                                    │                        │
            │                              room exit ─────────────────────┘
            │                              (branch below)
            │
   ── cross-cutting ──────────────────────────────────────────────────
   PAUSED      : app backgrounded; on resume, state = fold(event-log) (free)
   ABANDONED   : problem left / long idle; logged as a signal, not a failure
   DECAY       : a spaced retention probe on a MASTERED node fails → demote
                 (clears transfer_passed, lowers P_known) → node can re-open
```

**Room-exit branch** (when the gate passes):

1. If the room was entered via a kitchen wall → **ReturnToKitchen on the exact
   stumping recipe** (the philosophy doc's felt payoff; also a built-in
   near-transfer check).
2. The kitchen then presents the **next harder recipe**. If that recipe needs a
   not-yet-mastered skill → next `WALL_HIT` → next room. If it needs only
   mastered skills → the child advances the curriculum in the kitchen itself.
3. If the room was entered via **diagnostic skip-ahead** (no pending kitchen wall,
   e.g. confirming a strong child already has `ADD_SAME_DEN`) → advance directly to
   the next unmastered node or back to the kitchen, whichever the brain selects.

Plain-language walk-through of one full loop (kitchen → R2 → kitchen):

1. **KITCHEN, match-a-height** (the opener, also the cold-start probe). Mom asks
   for 3/4 cup; a target line appears on the measuring stick. Child stacks 1/4
   blocks to the line. Pure eye-matching, no arithmetic. Plants the goal, and
   collects baseline timing *before any wall logic runs* (this is the cold-start
   answer in §4.1).
2. **KITCHEN raises the bar: predict-the-sum.** Mom hands two piles, asks for the
   combined total *in fraction form before stacking*. Eye-matching no longer
   works. If a required skill is not yet mastered → `WALL_HIT`.
3. **Wall diagnosis** (§5): which skill is binding. First wall = `ADD_SAME_DEN`
   (the root skill); an unlike-base recipe makes `ADD_UNLIKE_DEN` binding. The wall
   routes to the most upstream unmastered node among the recipe's required skills.
4. **ROOM.** Drill that one skill with deliberately unfamiliar bases (5ths,
   7ths) so it is real addition, not recall. Scaffold fades within the room (§3).
   Stay until the **mastery gate** passes (§4.5).
5. **RETURN.** Child is dropped back into the kitchen on the exact stumping recipe.
6. **Scaffold fade across loops.** As nodes are mastered, kitchen recipes return
   with fewer blocks and shorter sticks, then bare numbers, then paper. The
   structure moves out of the game and into the child's head.

### Problem machine (one problem, in kitchen or room)

```
PRESENT ─► OBSERVING ─► (SELF_CORRECTING)* ─► SUBMITTED ─► JUDGED ─► (policy call, §5)
   │            │                                            │
   │      (idle / hint)                              correct / incorrect
   │            ▼                                            │
   │       HINTING ◄──────────────────────────────────────── ┘
   │            │
   └────────────┴──► ABANDONED  (long idle / app close; terminal, logged)
                                                    feeds the event log → model
```

- `PRESENT`: a problem appears (problem_generator) at the current scaffold level.
- `OBSERVING`: the child acts. Every pick/place/remove/idle is an event.
- `HINTING`: entered by request or by a Tier-2 heuristic (long pause,
  oscillation). Hints are a ladder (§3.1); the rung reached is recorded.
- `SUBMITTED → JUDGED`: verified deterministically (exact match against the
  generator's known answer). Correctness + latency + scaffold level + hint rung
  all written to the event log.
- `JUDGED → policy call`: the brain (§5) is consulted **only here**, at a problem
  boundary, never mid-attempt (premise 5).

## 3. Scaffold Ladder (the fade is a first-class axis)

Each room defines an ordered ladder. Lower scaffold = fewer supports = a harder
problem for the same skill. Example for `ADD_SAME_DEN`:

```
L0  full blocks, both piles visible, snap-to-grid, target line shown
L1  blocks visible, but the child must name the total before stacking
L2  "ghost" blocks (outlines only), child fills the number
L3  numbers only, no blocks            ← scaffold-independent
L4  numbers, novel surface form        ← transfer
```

A child correct only at L0 has not mastered the skill; they have mastered
*reading the blocks*. The ladder is how we tell those apart. This is the brief's
"fade scaffolds and test transfer once the learner seems ready," made concrete.

### 3.1 Scaffold transition logic (entry, step-down, step-up)

- **Entry level.** On first entry to a room, start at **L0**. On re-entry (after
  decay or a later wall), start one level below `max_scaffold_passed` from the
  node's `MasteryEstimate` (a small safety cushion), floored at L0.
- **Step down (fade) — harder.** After a correct streak of `k` (default 3) at the
  current level with latency in band and `hint_rung == 0`, the brain may
  `FadeScaffold` to the next level. Fading is a *decision the brain makes* (§5),
  bounded by these preconditions; the deterministic fallback fades exactly when
  the preconditions hold.
- **Step up (re-scaffold) — easier, work preserved.** After `m` errors (default 2),
  a stall (Tier-2 detects idle/oscillation), **or corroborated affect-distress**
  (§6.1: camera reads frustration/shutdown *and* a behavioral signal agrees),
  restore the next level of support **without discarding the child's current work**
  (brief: "offer a smaller hint while preserving the learner's work").
- **Why the LLM is allowed to deviate.** The brain may fade faster for a learner
  the camera reads as bored-and-fluent, or hold longer for one read as
  anxious-and-struggling, but only within the legal ladder, never past the
  deterministic gate, and only when affect is corroborated. The chosen level and
  the reason are logged for the "why did it change" orientation requirement.

### 3.2 The hint ladder (rungs, and how dependence is measured)

Hints are a separate, fixed ladder, recorded per problem as `hint_rung`:

```
H0  no hint
H1  attentional   ("look at the bottom numbers — are they the same?")
H2  procedural    ("make the bottoms match first")
H3  worked step   (one step done for them, child finishes)
H4  full solution (shown, then a fresh isomorphic item is required)
```

`hint_dependence` = fraction of recent correct answers that needed `hint_rung ≥ H2`.
It is not a free-floating field: it **gates** scaffold-independence and transfer
(§4.3, §4.4 require those successes at `hint_rung == 0`).

## 4. Student-State & Mastery Model (moved → measurement doc)

The full model — BKT accuracy, fluency, scaffold-independence, transfer, the gate,
decay, and the factorial-HMM upgrade — lives in **`student-state-measurement.md`**
(sections keep the same §1.1 / §4.x / §6.1 numbers). The load-bearing facts the rest
of *this* doc depends on:

- **`MasteryEstimate` is the interface.** Per node: `{ P_known, fluency_stats,
  max_scaffold_passed, transfer_passed, hint_dependence, last_retention_probe }`;
  student state is `{ node_id → MasteryEstimate }`. The measurement doc produces it;
  everything here reads it.
- **Mastery is multi-dimensional and the gate (§4.5) is deterministic** — the LLM
  brain can never set it (this is the §5.2 guardrail):

```
MASTERED(node) ⟺ P_known ≥ 0.95
              AND Scaffold-independent   (≥2 correct at L3+, hint-free)
              AND Transfer passed        (≥2 distinct surface forms, hint-free)
              AND Fluency OK             (soft pre-calibration; hard after)
```

- **Affect is never in the gate** — it changes pacing and support only
  (measurement doc §4.5/§4.7.3; this doc §6.1).
- **Decay (§4.6):** a `MASTERED` node gets spaced retention probes; failing one
  demotes it and lets the next wall re-open it.
- **Upgrade path (§4.7):** the four dimensions are the 2-state-BKT *floor*; a staged
  factorial-HMM upgrade (a `misconception` state, a vector emission, and
  strategy/engagement chains) swaps in behind the same `MasteryEstimate` — same
  "swap one implementation of one interface" discipline as the brain (§9).

## 5. The Tier-3 Brain (deterministic policy over structured state)

Per D5 (amended), the brain is a **deterministic policy** (§5.4). It is bounded by the same
guardrails that would have caged an LLM — legal-move enumeration and the exact verifier —
which now simply define a correct, inspectable policy. The brief's hard lines hold by
construction: content correctness (verifier), no false-positive mastery (the §4.5 gate), no
hidden uncertainty (recorded rationale).

### 5.1 The interface

Called only at problem boundaries (`JUDGED → policy call`):

```
nextDecision(studentState, skillGraph, legalMoves, recentBehavior) -> Decision + rationale

Decision ∈ { PresentProblem{node, scaffold, surface_form},
             RouteToRoom{node}, ReturnToKitchen{recipe},
             FadeScaffold, RaiseScaffold, TransferProbe{node},
             EscalateToHuman{reason, handoff_packet} }   // §5.5
```

Note what is **not** in the enum: there is no `DeclareMastered`. Mastery is
computed by §4.5, not chosen by the brain. The brain decides what to *present and
route*; the deterministic gate decides what is *learned*. `EscalateToHuman` is the
one decision the brain may *propose* but cannot *invent the grounds for* — like
mastery, its trigger is deterministic (§5.5).

`recentBehavior` carries the smoothed affect/attention state (§6.1) alongside the
behavioral signals. The brain may use "looks frustrated but close" to choose a
gentler legal move, or "bored and fluent" to fade faster — but every such move is
still drawn from `legalMoves`, still passes the verifier, and still cannot touch
the gate. Affect widens *which legal move feels right*, never *what is allowed*.

### 5.2 Deterministic guardrails (non-negotiable)

1. **Legal-move enumeration.** A deterministic function computes `legalMoves` from
   the skill graph and current state (e.g. you cannot `RouteToRoom` a node whose
   binding-ness isn't supported by the estimates; you cannot `FadeScaffold` past
   the gate's evidence). The LLM picks **from this set only**. An illegal or
   malformed response falls back to the deterministic policy (§5.4).
2. **Content correctness.** Every generated artifact (problem, hint, transfer
   item) passes the **same exact verifier** the deterministic generators use.
   The LLM may choose phrasing and which legal item to present; it may not invent
   math. Unverifiable artifact → rejected, fall back.
3. **Mastery is off-limits.** §4.5 is the only path to "mastered."
4. **Rationale is recorded.** The brain returns a one-line reason with every
   decision, surfaced to the child as "why did this change" and logged for
   inspection (counters "hiding uncertainty behind confident UI").

### 5.3 Wall detection & binding-skill diagnosis (deterministic input to the brain)

```
A recipe requires skill set S.
predicted_success = Π_{s ∈ S} P_known(s)
WALL_HIT  ⟺  predicted_success < θ   (default θ = 0.6)   OR   an actual attempt fails.
binding skill = the MOST UPSTREAM unmastered node among S
               (furthest from the goal / deepest foundation — i.e. ADD_SAME_DEN
                before ADD_UNLIKE_DEN). The brain then RouteToRoom it.
```

(Fluency is intentionally **not** folded into wall detection; it gates mastery,
not wall-firing. This removes the earlier undefined "discounted by fluency.")

### 5.4 The deterministic policy (the brain; also the harness baseline)

A complete rule-based `nextDecision` **is the brain**: route to the most upstream unmastered
prereq; `FadeScaffold` on a correct streak meeting §3.1 preconditions; `RaiseScaffold` on
errors; `TransferProbe` when the other dimensions are green; `ReturnToKitchen` on mastery.
It is deterministic, inspectable, and reversible, and it is the **baseline** the synthetic
harness measures against (§8).

### 5.5 Escalation to a human (the "out of moves" output)

The adaptive system is not allowed to pretend it always wins. When it has genuinely
exhausted its options, the right output is to **stop adapting and call a human**
(parent / teacher / tutor), not to keep churning the interface. This is a distinct
tutoring output, separate from the I/O modalities in §10 — it is the system
admitting the limit of automated remediation.

**Two trigger conditions** (deterministic, so the LLM can neither suppress nor
fabricate them):

1. **Stuck despite maximal remediation.** The system has broken the problem all the
   way down and the child still isn't progressing:
   - at the **floor scaffold** (L0) **and** the floor of the binding skill's
     prerequisite chain (already routed as upstream as the graph allows), and
   - `P_known` for the binding node has **not improved** over the last `N_stuck`
     attempts (default 6), with repeated errors even at `H3/H4` hints.
   In factorial-HMM terms (§4.7): Chain A stays pinned at `unknown`/`misconception`
   while every legal support is already applied. The machine has no smaller step left.

2. **Disengaged / not participating.** The child is not doing the work, so no amount
   of breaking-down helps:
   - sustained Chain-C `avoiding`/`bored` (§4.7.2), **corroborated by behavior**
     (idle, rapid-submit-without-working, attention `away`) per the §6.1
     affect-corroboration rule — affect alone never triggers escalation, and
   - over a window (`N_diseng` attempts / minutes), not a single bad moment.

**What escalation does (and does not) do:**

- **Pauses adaptation and hands off.** The game enters a calm "let's get some help"
  state rather than presenting another problem. It does **not** punish, fail, or
  lock the child out.
- **Produces a `handoff_packet`** for the human: which skill is binding, the recent
  attempt history, the scaffolds/hints already tried, the error signatures seen
  (stuck case), or the engagement trajectory (disengaged case), plus the suggested
  reason. This is the §4.7.4 event log made human-readable — the human starts
  informed, not cold.
- **Distinguishes the two reasons**, because they need different humans-in-the-loop:
  "stuck" is a *teaching* problem (the content/sequence may have a real gap — feed it
  to the §8 harness), "disengaged" is a *motivational/context* problem (maybe the kid
  is tired, not in the mood, or needs a person).
- **Is reversible.** After a human interaction the child resumes from the same state
  (state is a fold over the event log; nothing was lost).

**The LLM's role:** it may *raise* an escalation hypothesis earlier (e.g. it reads
mounting frustration) and it phrases the handoff, but the **deterministic trigger is
the only thing that fires the escalation** — symmetric with mastery. This protects
both directions: an over-eager LLM can't escalate gratuitously (annoying the human,
teaching the child to bail), and an over-optimistic LLM can't keep a drowning child
in the loop by refusing to escalate.

**Counter-metric (it must not over- or under-fire):** track escalation rate, the
false-escalation rate (human says "that child was fine"), and the *miss* rate (a
child who churned far past the point a human should have been called). Tuned against
the synthetic personas — the *anxious* and *adversarial/distractible* personas are
the direct probes (§8).

## 6. The Three Responsiveness Tiers (mapping to `hyper_responsive_ui`)

| Tier | Latency | Reads | Authority | Examples |
|------|---------|-------|-----------|----------|
| **T1 Reflex** | ms | the current physical action only | UI, deterministic, no model | block snaps to grid; stack glows at the line; gentle shake on overshoot |
| **T2 Heuristic** | seconds | a sliding window of recent events, incl. smoothed affect | rule-based; *nudges only* | long pause → offer next hint rung; place/remove oscillation → "take your time, what's the total?"; too-fast-correct → quietly queue a transfer probe; corroborated frustration → soften / re-scaffold |
| **T3 Strategic** | between problems | full student state + affect summary | the §5 brain; the only tier that changes *what you're doing* | route to a room, fade scaffold, return to kitchen, transfer probe |

T1 and T2 are deterministic and instant. Only T3 calls the LLM, and only at
problem boundaries, so seconds-latency is acceptable. This table is the brief's
"which signals the system listens to, ignores, or treats cautiously" and "when
the UI should change automatically vs wait."

### 6.1 The affect/attention signal (summary; full detail in measurement doc §6.1)

The camera emits a **smoothed** `AffectState { engagement, valence, attention,
confidence }` at ~1–2 Hz, as Signal events (§1.1). The rules that keep it from being
a gimmick — **corroboration before action** (affect alone triggers nothing; a
behavioral signal must agree), **never gates mastery**, **on-device, no raw video**,
and **must beat behavior-only or it gets cut** — and how it maps to the pedagogy
(productive-struggle vs. shutdown) live in **`student-state-measurement.md` §6.1**.
What matters for the tiers: affect is read by T2 (nudges) and T3 (the brain), never
by T1, and never by the gate.

### False-positive guards

Guessing, pattern-matching, misconception, hint-dependence, and click-through/
avoidance each map to a specific recorded signal and mastery dimension; the full
table is in **`student-state-measurement.md` → False-positive guards**. The synthetic
personas (§8) are the probes for each.

## 7. Stability Policy — what the interface refuses to change automatically

- **T1 is always on** (it is the feel of the game).
- **T2 only nudges** (hints, encouragement); it never restructures the workspace.
- **T3 acts only at problem boundaries**, never mid-attempt.
- **A key visual stays stable** while surrounding guidance changes (the measuring
  stick and current blocks do not rearrange when a hint appears).
- **Escalation is a boundary action too.** `EscalateToHuman` (§5.5) fires only at a
  problem boundary, pauses adaptation calmly, and preserves the child's work — it
  never yanks the child mid-attempt and never reads as failure/punishment.
- **Refuses to:** pull a child out of a room mid-attempt; auto-advance past an
  unmastered binding skill; switch modality without the child initiating;
  **declare mastery on completion alone** (only §4.5 can); let the LLM change
  state, generate unverified math, or fade past the gate's evidence; **act on
  affect alone** (§6.1 requires behavioral corroboration) or treat affect as
  evidence of knowing.

## 8. The synthetic_challenger Seam (designed-in now, built later)

`synthetic_challenger` is a **red-team harness of fake students**, not a content
generator. It plugs into the same engine (premise 1).

```
Persona (policy) ──emits actions──► [ENGINE] ──event log + mastery trajectory──►
   │                                                                            │
   └──────────────── failure analysis ◄── cluster failures ◄───────────────────┘
                            │
                edit brain prompt / policy params / skill graph
                            │
                     re-run population → regression check vs held-out personas
```

- **A persona emits the same action AND signal events a child would** (§1.1),
  governed by a behavior policy — including **synthetic affect**. Mapped to the
  brief's list: *confident guesser* (fast submit, delighted valence, fails
  transfer), *memorizer* (reaches `P_known ≥ 0.95` but fails every
  `TransferProbe`), *over-hinter* (requests hints before attempting, never reaches
  `hint_rung == 0` at low scaffold), *shortcut-seeker* (submits without stacking),
  *anxious* (frustrated valence + early abandon — the persona that probes whether
  affect-driven re-scaffolding actually helps), *distractible* (attention drifts /
  abandons when scaffold rises). The *anxious* and *distractible* personas, plus a
  dedicated *immovably-stuck* persona (never progresses on the binding skill no
  matter how far it's broken down), are the direct probes for the **escalation
  trigger (§5.5)** — they should fire it, and a *normal-but-slow* persona should
  **not** (false-escalation check).
- **What it tests:** the curriculum, the §4.5 gate, AND the LLM brain. Because D5
  put an LLM in the loop, the harness has extra jobs the brief explicitly names:
  "when does the tutor give away answers instead of teaching" (does the LLM jump
  to H4 too fast for the over-hinter?) and "which changes improve learning vs
  merely improve the benchmark" (LLM brain vs the §5.4 deterministic baseline).
- **Overfit guard (the brief's hardest requirement):** hold out a subset of
  personas; require a human-plausible failure before accepting a fix; check that a
  fix helping one persona does not regress another.

Same fold, same code path for real and synthetic → the synthetic evidence is
credible rather than a divergent simulator.

## 9. Implementation Note: how the brain stays safe (deterministic)

D5 chose the most flexible and the riskiest brain. The safety comes entirely from
§5.2: the LLM is a *chooser among deterministically-legal, deterministically-
verified moves*, never an authority on truth or mastery. Concretely:

- It cannot emit illegal moves (enumerated set).
- It cannot present wrong math (exact verifier on every artifact).
- It cannot declare mastery (deterministic gate).
- It cannot act mid-attempt or change a stable key visual (P5).
- If it is slow, down, or malformed, the deterministic policy (§5.4) takes over
  invisibly.

This is what lets you claim, against the brief, that the AI-native brain does not
"hide uncertainty behind confident UI": every LLM decision is legal, verified,
reasoned, and reversible to a deterministic baseline.

## 10. Modality Layering

Every modality is an action-emitter, a signal-source, or a render-target on the
engine (premise 1; §1.1). Build order (everything ships; this is dependency order,
not a cut):

1. drag-to-stack + tap/type answer (action input); visual workspace + animated
   feedback (output). The first running loop.
2. + Spoken tutor (output): TTS for mom's prompts and feedback.
3. + Voice prediction (action input): say the sum before stacking. High
   pedagogical value (verbal pre-commitment is a strong anti-pattern-match signal).
4. + Handwriting (action input): write the fraction, recognized via ink/camera;
   the workspace preserves the child's handwritten work (a brief flagship example).
   Handwriting serves **two** distinct purposes (see §10.1).
5. + Affect/attention camera (**signal** input, §6.1): on-device emotion /
   engagement / attention, smoothed, corroborated, never in the mastery gate.
   Independent of the action inputs, so it can land in parallel with 2–4; it only
   feeds the model and the Tier-2/3 brain, so it cannot destabilize the game loop.

### 10.1 Handwriting has two jobs (answer entry vs. symbol-binding)

1. **Answer entry** — at low scaffold the child writes the resulting fraction
   instead of tapping it. A normal action input; recognized, verified, judged.
2. **Symbol-binding (memory-encoding aid)** — the child writes the *numeral* next
   to a concrete stack of blocks, by hand, to bind the abstract symbol to the
   quantity it stands for. This is the philosophy doc's "translate the block
   representation to the numeric representation" sub-skill, and it is the
   memory-palace thesis applied: the handwritten symbol is an anchor connected to
   a vivid concrete thing (the stack), so the count-↔-number association sticks.
   It is most useful at the **L1/L2 scaffold rungs** of every room, where blocks
   and numerals coexist and the act of writing the number does the binding. Pedagogically this is *why* handwriting beats tapping for early
   encoding: forming the glyph by hand is a richer, more durable anchor than
   selecting it. (Tapping is fine once the association is already automatic.)

Both jobs use the same handwriting input modality and the same exact verifier;
they differ only in *when* the brain presents them and *what* the recognized glyph
is checked against (the answer, vs. the count of blocks currently stacked).

## Success Criteria & Counter-Metrics (from the brief)

**Primary**
- A child moves from a felt wall to a defended mastery signal on ≥2 skills.
- The §4.5 gate passes only when all dimensions hold (fluency soft pre-calibration).
- Skip-ahead demonstrably fires for a strong learner who already has an upstream
  room (e.g. `ADD_SAME_DEN`), routing them past it.
- A transfer moment exists per node and is logged.
- Every T3 change carries a recorded, child-visible rationale.

**Counter-metrics (protect against bad responsiveness / shallow learning)**
- **UI churn:** count of T3 changes per session; alarm if it changes too often.
- **Orientation:** can the child always state goal / next action / why it changed?
- **Dependence:** unassisted-correct rate at low scaffold trending up, not hint
  rate up.
- **False-positive mastery rate:** % of mastered nodes failing a delayed probe (§4.6).
- **LLM-brain value:** does the LLM brain beat the §5.4 deterministic baseline on
  learning gain and transfer, or just on engagement? (the brief's "improve
  learning vs improve the benchmark").
- **Answer-giving:** rate at which the LLM jumps to H3/H4 hints (synthetic
  over-hinter persona is the probe).
- **Escalation calibration (§5.5):** escalation rate, false-escalation rate (human
  says the child was fine), and miss rate (child churned far past where a human
  should have been called). The stuck/anxious/distractible personas should trigger
  it; the normal-but-slow persona should not.
- **Affect signal value:** does affect-driven pacing beat behavior-only on
  learning outcomes and on shutdown-recovery (does the anxious persona, §8,
  actually recover more often with affect on)? If affect does not beat
  behavior-only, the camera gets cut (§6.1). Track affect noise vs. usefulness.
- **Baseline comparison:** the same skills via a static block-stacker, no
  adaptation — time-to-mastery and false-positive rate.
- **Model-upgrade value (when §4.7 is built):** does the richer HMM (misconception
  state, vector emission) actually lower the false-positive-mastery rate and catch
  misconceptions earlier than the 2-state BKT floor — or just add parameters? The
  harness-calibrated model must beat the BKT baseline on held-out personas, not the
  ones it was tuned on (§4.7.5).

## Open Questions

1. **Age band for fluency.** What latency target per skill counts as fluent? Needs
   a pilot; fluency is a soft gate until then (§4.2).
2. **How fine to split sub-skills.** Start coarse (4 nodes); split `SIMPLIFY` into
   factor-finding / shared-factors / divide only when a wall routes there.
3. **LLM call cost/latency at problem boundaries.** Acceptable per P5, but measure;
   cache legal-move enumeration; consider a small fast model for routing and a
   larger one only for hint phrasing.
4. **Transfer-form library size.** How many structurally distinct forms per node
   before the gate is trustworthy? The synthetic memorizer persona is the test.
5. **Affect model for children.** Off-the-shelf affect models are trained mostly
   on adults; child facial affect differs. What accuracy can we get on-device, and
   what is the false-frustration / false-boredom rate? Until measured, affect stays
   low-trust and corroborated (§6.1); the camera is opt-in.
6. **Credit assignment across the DAG (§4.7.4 step 3).** When a wrong answer could
   implicate the binding node or a prerequisite, how is the update split? First-pass
   proposal: update the binding node; propagate a discounted update to a prereq only
   when the `error_signature` implicates it. Needs validation via the harness.
7. **Error-signature library.** How many misconception signatures per skill, and how
   are they authored/verified? The room docs already name the main ones; the
   featurizer (§4.7.4) needs them enumerated and tagged.
8. **Escalation thresholds and channel (§5.5).** What `N_stuck` / `N_diseng` values
   actually separate "needs a human" from "needs one more attempt" — tune on the
   synthetic personas, validate with real kids. And concretely, who is escalated to
   and how (in-app prompt to a present parent, a flag in a teacher dashboard, a
   tutor handoff)? The trigger is defined; the delivery channel is product-dependent
   and unspecified.

## Risks

- **Scope (highest).** Four rooms + five modalities + LLM brain + harness is large.
  Mitigation: strict dependency order (M0→M6); each milestone is independently
  demoable, so the demo degrades gracefully if later milestones slip.
- **LLM brain correctness/latency.** Mitigation: the §5.2 cage + §5.4 fallback;
  the LLM never owns truth or mastery.
- **Fluency calibration without real kids.** Mitigation: soft gate + synthetic-
  harness sensitivity check.
- **Model over-reach (§4.7).** The richer factorial HMM has many parameters we
  cannot calibrate without data; a sophisticated-looking but mis-calibrated model is
  exactly the brief's "hide uncertainty behind confident UI" failure. Mitigation:
  ship the 2-state BKT floor; gate each upgrade on beating the floor on *held-out*
  personas (§4.7.5–4.7.6); keep the `MasteryEstimate` interface stable so a bad
  upgrade reverts to the floor with no downstream change.
- **Modality input error (voice/handwriting).** Mitigation: last in build order;
  the engine seam means they cannot destabilize earlier milestones.
- **Affect over-trust / noise.** A noisy or adult-biased affect model could push a
  fine child into needless re-scaffolding, or read a bored kid as engaged.
  Mitigation: affect is a signal not an action (§1.1), never gates mastery (§4.5),
  acts only with behavioral corroboration (§6.1), and is kept only if it beats
  behavior-only on the counter-metric. Privacy mitigation: on-device, no raw video.
